3x4 模式接入清单

一、现状结论

当前系统里，“新增比赛模式”至少涉及四层：

1. 模式目录
services/competition_mode_service.py

2. 建赛默认规则
services/event_admin_service.py

3. 结算取分逻辑
services/settlement_service.py

4. 基础字典
config/variants.json
config/rating_buckets.json

其中最关键的一点是：

event_rule_sets.ranking_metric 目前还不是唯一真源。
settlement_service.metric_from_record() 仍会按 competition_type 决定主指标取值。

这意味着如果 3x4 的主指标不是：

- classic/fibonacci 的 raw_score/final_score
- stone/no_x 的 competition_score
- speedrun 的 primary_time_ms

那就不能只加一个模式枚举，必须同步扩展结算逻辑。

二、3x4 需要先拍板的四个问题

1. 平台来源

可选方向：

- A. 2048verse
  适合接 Verse 现有账号体系、报名体验、锁局流程。
  但前提是 Verse 真的有可稳定识别的 3x4 成绩来源。

- B. Taihe
  适合走“导入成绩文件 / Bot 手填待审核”路线。
  当前 Stone / No-X / Speedrun / Fibonacci 都是这条路径。

- C. 其他平台
  需要先补 platforms.json，必要时再补导入器或附件解析规则。

建议：
如果 3x4 没有成熟的自动识别接口，先按 Taihe/导入型模式落地，风险最低。

2. 排名主指标与排序方向

需要明确最终排名到底看哪个字段：

- raw_score / final_score
- competition_score
- primary_time_ms
- 其他自定义值

排序方向：

- desc：越高越好
- asc：越低越好

建议：
如果 3x4 是“原始分赛”，尽量沿用 raw_score/final_score + desc。
如果 3x4 是“达成耗时赛”，用 primary_time_ms + asc。
如果是平台特制分，明确写成 competition_score，避免和 raw_score 混淆。

3. 多局聚合方式与缺局策略

系统现有可直接支持：

- best_single
- best_of_n
- average_of_n
- sum_of_best_n

缺局策略当前只对 average_of_n 有现成语义：

- missing_attempt_policy = zero_fill

需要明确：

- 是单局赛，还是多局赛
- 如果多局，是取最好、平均，还是求和
- 不满局数时，是按 0 计、按已完成局数计，还是直接不出成绩

建议：
如果想减少争议，优先选：

- 单局最好 best_single
- 多局平均 average_of_n + zero_fill

这两条都和现有代码路径最贴近。

4. 是否需要锁局（floor / finish）及证据要求

当前锁局链只稳定服务于 2048verse：

- floor：上传 early 回放
- finish：上传 final 回放
- 前缀校验：early 必须是 final 前缀
- attempt_session 自动推进 completed

需要明确：

- 3x4 是否真的需要“开局到终局”的证据链
- 平台是否能稳定产出可校验回放
- 证据是否必须前缀一致
- 不走锁局时，是否只要求成绩文件 / 截图 / 手工审核

建议：
如果 3x4 不是 Verse 且没有标准回放文件，先不要硬接 floor/finish。
先走“成绩提交 + 待审核”模型更稳。

三、落地映射

如果最终方案是“导入型 3x4 模式”，通常需要改这些点：

1. config/variants.json
新增 3x4 variant。

2. config/rating_buckets.json
新增对应 bucket。

3. services/competition_mode_service.py
新增 3x4 模式定义。

4. services/event_admin_service.py
补 default_rule_values() 和 infer_rating_bucket_code() 映射。

5. services/settlement_service.py
确认 metric_from_record() 能正确取到 3x4 的排名主指标。

6. importers/raw_score_file_parser.py
如果成绩字段和现有 raw_score / competition_score / primary_time_ms 不同，需要扩展解析。

7. Bot / CLI
如果要让选手从 QQ Bot 直接提交 3x4 成绩，要确认输入口令和审核提示文案。

如果最终方案是“Verse 锁局型 3x4 模式”，还要额外确认：

1. 平台账号绑定是否仍复用 2048verse
2. attempt_service 是否允许该 variant 启动锁局
3. replay_lock_service 的证据校验是否适用于 3x4 回放
4. Bot 的 floor / finish 提示文案是否要按模式区分

四、推荐的最小可行方案

优先建议先做这一版：

- 平台：Taihe 或导入型平台
- 模式类型：3x4 原始分
- 排名主指标：raw_score 或 final_score
- 排序方向：desc
- 聚合方式：best_single
- 缺局策略：无
- 锁局：不启用
- 证据：成绩文件导入，必要时 Bot 走待审核

这样改动最小，基本可以复用 Fibonacci/Stone 一类模式的接入路径。

五、如果要做“显著不同规则”的 3x4

当出现以下任一情况时，建议不要只补配置，应该顺手做一次小重构：

- 排名指标是新的字段
- 同一模式下既要看分数又要看时间
- 聚合规则不是现有四种
- 缺局策略不是 zero_fill
- 锁局条件因模式不同而变化

推荐重构方向：

1. 让 competition_mode_service.py 显式声明：
- ranking_metric
- ranking_order
- validation_rule
- rating_bucket_code / bucket 映射策略
- score_field_hint

2. 让 settlement_service.py 优先按 rule_set.ranking_metric 取值，
而不是主要依赖 competition_type 写死分支。

这样之后再加 3x4 变体、6x6、特殊计分赛，成本会低很多。

六、等待确认的问题

请至少确定以下四项：

1. 3x4 来自哪个平台
2. 最终排名主指标是什么，越高还是越低更好
3. 单局还是多局，多局如何聚合，缺局怎么算
4. 要不要 floor / finish 锁局，证据文件是什么格式
