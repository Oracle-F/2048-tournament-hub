# 事故时间线索引

本文档用于索引项目中的重要事故、止损、修复和观察过程。

设计目标：

* 不让单个事故记录文件无限膨胀
* 让新 Agent 能快速找到最近的事故脉络
* 让月度明细文件保持可维护、可归档

## 使用方式

* 这里不写所有细节，只保留索引和拆分规则。
* 具体时间线写在对应月份文件中，例如：
  * `docs/incident_timeline/2026-05.md`
  * `docs/incident_timeline/2026-06.md`
* 如果事故跨月，优先按“首次发生时间”归入对应月份。
* 如果某月事故特别多，可以继续拆成更细粒度文件，但要保持索引更新。

### 索引字段说明

* 时间：事故首次被确认的日期

* 月度文件：事故详细时间线所在文件

## 拆分规则

### 什么时候新建月度文件

* 新事故首次发生在新月份。
* 旧月已有多次事故，导致文件阅读困难。
* 需要把一个月内的事故单独沉淀出来。

### 什么时候继续写当前月文件

* 事故还在同一个月内持续发生。
* 同一问题出现新的证据、修复动作或观察结果。
* 只是同一事故的补充排查，不需要新建条目。

### 什么时候拆更细

* 单月事故过多，文件明显过长。
* 某一类事故特别密集，适合单独分组。
* 需要把“Bot / DB / Hub / 计分”拆开维护时。

可选拆分方式（按需创建）：

* `docs/incident_timeline/2026-05-bot.md`
* `docs/incident_timeline/2026-05-db.md`
* `docs/incident_timeline/2026-05-hub.md`

这些只是按主题继续拆分的示例文件名，只有在某个月事故明显增多、确实需要分流维护时才创建。

## 归档规则

* 当前月文件在月底后不再高频改动。
* 如果某个月的事故已稳定且不再变化，可视为封存。
* 旧文件不要删除，保留为历史参考。
* 若某事故已从 `bug_history.md` 迁移到 `bug_archive.md`，时间线里仍可保留简短记录，但不要重复展开细节。

## 与其他文档的关系

* `bug_history.md`
  * 当前持续关注的问题清单
* `bug_archive.md`
  * 已归档问题清单
* `incident_timeline.md`
  * 时间线索引和拆分规则
* 月度事故文件
  * 记录具体过程、止损、修复和观察
* `maintenance_rules.md`
  * 规定这些文档如何维护
* `agent_handoff.md`
  * 给新接手 Agent 的快速入口

## 维护原则

* 索引文件保持短，月度文件承载细节。
* 事故标题尽量固定，便于全文检索。
* 观察状态改变时，优先更新索引和对应月度记录。
* 不要把所有历史都堆进一个大文件。
* 如果后续发现更合理的拆分方式，可以继续细化，但要同步更新索引。
