- 某些统计为了防止假 0，已经加了“抓不全则不乱回 0”的思路，但这也意味着在高压下可能出现“稍后再试 / 繁忙”体验。

七、并发、锁冲突、NapCat 链路：当前最新判断

这是 2026-05-29 最需要接着收的主问题。

1) 已暴露出的根问题
- SQLite 单库并发写脆弱
- Hub 重刷新和 bot 实时交互共享同一个库
- 以前 bot 整条消息还被包在大事务里，已收紧
- 错误兜底曾经会在群里 `@人 处理失败：database is locked`，已止血
- NapCat / OneBot 链路本身也会抖，尤其 `get_msg timeout`、`sendMsg Timeout`

2) 已做的并发治理
文件：赛事中台\db.py
- connect 增加 timeout
- PRAGMA foreign_keys = ON
- PRAGMA busy_timeout = 10000
- 尝试 PRAGMA journal_mode = WAL（失败则降级，不阻止启动）

文件：赛事中台\bot_private_qq\app.py
- 不再每条消息都 `initialize_schema(connection)`
- 改成进程级一次性尝试 schema init
- schema init 若因锁竞争失败，不阻止启动
- 不再把整条消息处理包在 `with transaction(connection):` 里
- SQLite 锁库类错误：
  - 群聊静默
  - 私聊回 `系统繁忙，请稍后再试。`
- 发送层 `sendMsg Timeout`：单独记日志，不再放大成群刷屏
- 消息处理已加短暂自动重试：
  - 锁库时重试 3 次
  - 间隔约 0.35 秒

文件：计分器\organizer_event_hub.py
- Hub Verse 手动刷新改成：
  - 先抓网络
  - 再按玩家短事务写库

3) 当前并发层已知/潜在问题
- bot 现在能启动，但在某些时刻仍可能私聊返回：`系统繁忙，请稍后再试。`
- 这说明锁竞争仍偶发存在，只是已经从“炸群刷屏”降级成“短暂繁忙”。
- 如果用户在刷新后很久才发 `floor` 仍繁忙，说明不一定是刷新当下，也可能有别的进程长时间持锁。
- 需要继续排查是否存在：
  - 两个 bot / hub 实例并存
  - 某条服务逻辑自己长时间持锁

4) NapCat / OneBot 侧暴露出的新问题
日志已出现：
- `Error when getting message reply info: WebSocket call api get_msg timeout`
- `WebSocket for Bot ... closed by peer`
- `sendMsg Timeout`

这说明：
- 慢和异常不只来自 SQLite
- 还来自 NapCat / OneBot 链路本身
- 尤其是 reply / 引用消息会拖慢处理

对下一对话的建议：
- 不要只盯 SQLite
- 也要盯 reply/get_msg 超时和 WebSocket 抖动

5) 锁局赛 bug：未锁局普通局可能混入成绩，已修
根问题：
- 锁局赛的排名/结算读取 `event_attempt_records` 时，之前没有严格限制“必须是锁局有效记录”。
- 因此 `verse_manual_refresh` 之类的普通候选局，只要进了赛事记录表并在时间窗内，就可能混进锁局赛排名。

当前修复：
文件：赛事中台\services\settlement_service.py
- 锁局赛现在只允许以下 `record_type` 参与排名/结算：
  - `locked_first_game`
  - `tracked_next_record`
  - `manual_score_entry`

因此：
- 普通 `verse_manual_refresh` 记录不再参与锁局赛排名
- 这条修复是结果读取层的硬过滤，不是仅在 UI 层提示

- 重点关注：私聊 floor / finish 在非刷新时段仍繁忙的根因。

P2：NapCat / OneBot 链路
- 继续处理 `get_msg timeout`、`sendMsg Timeout`、WebSocket 被 peer 关闭的问题。
- 尽量减少对 reply 信息的依赖，或者让 reply 超时快速降级。

P3：Verse 查询精修
- 现在大方向已改，但很多统计仍需按“具体玩家 + 具体命令”继续钉。
- 尤其：16ks / 32ks / 8/16 / 16/32 / 1024s / 32k综率。

P4：群聊流程大测
- 重点测：
  - 群聊 提交成绩
  - 群聊 floor
  - 群聊 finish
  - 多人同时在一个群跑流程
  - 引用消息 / @ / 纯文本提及

十一、给下一次对话的最短结论

当前项目已经从“私聊赛事 bot”扩成了“Hub + 私聊 + 群聊 + Verse 查询”的一整套系统。

功能面已经很全，但 2026-05-29 晚上的主问题不再是“有没有这个功能”，而是：
1. SQLite 锁竞争
2. NapCat / OneBot 超时与 WebSocket 抖动
3. 个别 Verse 统计仍需逐玩家精修
