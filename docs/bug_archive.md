# 归档记录

## 2026-06-03 | QQ 私聊 bot 启动依赖缺失导致无法再次启动

* 模块：QQ 私聊 bot / 启动脚本 / Python 运行环境
* 风险等级：低
* 现象：
  * bot 入口脚本启动时会因缺少 `nonebot` 相关依赖而失败。
  * `run_bot.bat` 与自动启动脚本指向的 Python 环境不能稳定完成 bot 导入。
  * 结果是私聊 bot 无法重新拉起。
* 根因：
  * 启动所用的 Python 解释器没有装齐 bot 依赖。
  * 启动链路需要和实际安装依赖的解释器保持一致，否则入口会直接在 import 阶段报错。
* 修复：
  * 将 `requirements-bot.txt` 安装到实际启动使用的 Python 3.12 环境中。
  * 统一修正 `run_bot.bat` 与 `scripts/setup_bot_autostart.ps1`，让它们使用同一个可用解释器。
  * 重新验证 bot 启动脚本的导入链路可以正常完成。
* 状态：已归档
* 相关文件：
  * `赛事中台/run_bot.bat`
  * `赛事中台/scripts/setup_bot_autostart.ps1`
  * `赛事中台/requirements-bot.txt`
  * `赛事中台/bot_private_qq/app.py`
* 备注：
  * 这类问题不是业务逻辑故障，而是运行环境和启动入口没有对齐。
  * 后续如果换 Python 版本或重装环境，先确认 bot 依赖是否仍在同一个解释器里。

## 2026-06-02 | Codex Windows app 重置前启动异常与任务栏旧入口失效

* 模块：Codex Windows app / 启动入口 / 本地状态
* 风险等级：低
* 现象：
  * Windows 启动 Codex 时弹出 `Error launching app`。
  * 报错内容显示无法找到 `OpenAI.Codex_...` 对应的 Electron app，并把 `type=click&tag=...` 一类参数带进了启动路径。
  * 应用设置里的“修复”持续提示稍后重试，且跨重启仍然存在。
  * 任务栏固定入口清理后重新固定，说明旧 pin 也受到了影响。
* 根因：
  * 推测为更新后旧启动入口或应用注册信息未刷新。
  * 任务栏 pin 沿用了过期入口，导致深链参数被当成了包路径。
* 修复：
  * 先备份 `C:\Users\oracl\.codex`，避免本地状态误伤。
  * 使用 Windows 应用“重置”清除异常状态。
  * 重置后从开始菜单重新启动 Codex，并重新固定任务栏入口。
* 状态：已归档
* 相关文件：
  * `C:\Users\oracl\.codex`
  * `赛事中台/docs/incident_timeline/2026-06.md`
* 备注：
  * 这次故障未造成对话历史丢失，但说明 Codex Windows app 的启动注册状态可能会在更新后失效。
  * 后续若再次出现同类问题，优先检查旧入口和应用注册，不要先怀疑项目数据。

## 2026-05-31 | 测试赛长期残留导致数据库脏数据

* 模块：测试赛 / 成绩库 / 赛事状态管理
* 风险等级：中
* 现象：
  * 某些已经结束或不应再参与的测试赛记录，长期留在数据库里。
  * 这些残留数据可能干扰后续查询、补录和结算判断。
  * 有时会让新旧测试赛的状态边界看起来不清晰。
* 根因：
  * 早期测试赛主要是为了验证流程，后来累积成了历史包袱。
  * 清理和归档策略不够统一。
  * 某些查询默认把“还活着的测试赛”也算进了可见集合。
* 修复：
  * 梳理并修正测试赛的状态与归档口径。
  * 对历史残留数据做人工核对，必要时恢复为正确状态而不是直接删除。
  * 让查询和结算更严格地区分 active / cancelled / archived。
* 状态：已归档
* 相关文件：
  * `赛事中台/docs/项目现状总结_供新对话.txt`
  * `赛事中台/services/registration_service.py`
  * `赛事中台/services/settlement_service.py`
* 备注：
  * 这类问题是“测试环境变成生产历史”的典型后遗症。
  * 后续建赛时要尽量避免让测试赛长期残留为默认可见项。

## 2026-05-30 | 特殊计分逻辑边界漏洞

* 模块：锁局结算 / 特殊计分 / 成绩归档
* 风险等级：中
* 现象：
  * 某些特殊计分或锁局结算场景下，边界记录可能被错误纳入或排除。
  * 个别比赛会出现“明明像是有效记录，但结算口径不一致”的情况。
  * 早期还出现过特殊比赛的统计口径与普通局混用的问题。
* 根因：
  * 结算层对记录类型的边界约束不够严格。
  * 某些“看起来像有效局”的记录，实际上不属于应参与排名的集合。
  * 人工补录和自动抓取的来源标记没有完全统一。
* 修复：
  * 对锁局赛参与排名的记录类型做硬过滤。
  * 将普通刷新记录与锁局有效记录分离。
  * 对需要保留的记录，通过会话绑定和状态修正来恢复，而不是直接按表面特征删除。
* 状态：已归档
* 相关文件：
  * `赛事中台/services/settlement_service.py`
  * `赛事中台/docs/项目现状总结_供新对话.txt`
* 备注：
  * 这类问题最危险的地方在于“结果看起来像是对的，但口径其实已经错了”。
  * 新赛事上线时，最好先确认 record type 和结算规则的映射关系。
