# 2026-05 事故时间线

本月记录只保留已确认、对维护有价值、且后续可能影响稳定性或结果的问题。
一些一次性的边角异常、未确认的小波动、或没有形成明确维护结论的情况，不计入本文件。

## 2026-05-27 | Bot异常刷屏与账号风控警告

* 发现时间：2026-05-27
* 风险等级：高
* 触发入口：群聊 / 私聊 / 高风险消息
* 现象：
  * 机器人在群聊和私聊场景里出现过高频响应、连续回复和复杂消息发送。
  * 账号侧出现过风控警告或疑似风险升高的迹象。
  * 当时需要优先止损，而不是继续扩大测试范围。
* 止损动作：
  * 关闭群聊入口，回退到低风险可用态。
  * 取消默认 `@` 回复。
  * 暂停图片帮助、文件上传、成绩提交等高风险入口。
  * 只保留私聊轻命令作为最小可用面。
* 排查过程：
  * 先确认是否为单纯代码错误，随后发现更像是平台风控与链路抖动叠加。
  * 观察到 `input_status`、回复超时和发送抖动并存，说明问题不只是逻辑层。
* 修复动作：
  * 将机器人策略收回到保守模式。
  * 群聊和高风险能力保持关闭。
* 结果：
  * 当前维持低风险策略，不再把群聊作为常态能力。
* 后续观察：
  * 只允许低频、轻量、私聊优先的恢复节奏。

## 2026-05-27 | NapCat偶发 Kick Offline / WebSocket 抖动

* 发现时间：2026-05-27
* 风险等级：中
* 触发入口：reply / 引用消息 / 复杂消息段
* 现象：
  * 出现过 `WebSocket closed by peer`、`sendMsg Timeout`、`get_msg timeout` 一类异常。
  * 有时消息能进来，但回复明显变慢，甚至无法及时返回。
  * 某些情况下会伴随掉线或连接重建现象。
* 止损动作：
  * 对发送超时和链路异常做降级处理。
  * 尽量减少对 reply 信息的依赖。
  * 对文件补全和回复发送增加超时容忍与日志记录。
* 排查过程：
  * 发现问题不只来自业务逻辑，也来自 OneBot / NapCat 链路本身。
  * reply 与复杂消息段会明显放大延迟。
* 修复动作：
  * 将抖动视为链路基础问题，优先做降级而不是强行重试轰炸。
* 结果：
  * 链路仍可能抖动，但已尽量避免把抖动放大成群可见错误。
* 后续观察：
  * 后续维护要默认把它视为“可能随时抖动的基础设施”。

## 2026-05-29 | SQLite锁竞争导致消息处理短暂繁忙

* 发现时间：2026-05-29
* 风险等级：中
* 触发入口：Hub 重刷新 / bot 实时消息 / 共享写入
* 现象：
  * 私聊和群聊有时返回“系统繁忙，请稍后再试”。
  * Hub 的重刷新或其他后台任务运行时，bot 消息处理变慢。
  * 高并发阶段偶发数据库锁等待。
* 止损动作：
  * 增加连接超时与等待容忍。
  * 锁错误做短重试，并在用户侧降级为繁忙提示。
  * 缩短事务范围，把网络请求和写库拆开。
* 排查过程：
  * 先从 bot 返回“繁忙”入手，再逐步确认并不是单一命令的问题。
  * 对照日志后发现，锁竞争往往与刷新、导入或补录任务相关。
* 修复动作：
  * 让重操作更细粒度地写入，减少长事务持锁时间。
* 结果：
  * 已缓解，但高压时段仍需持续观察。
* 后续观察：
  * 若并发再上升，需重新评估单 SQLite 架构的上限。

## 2026-05-30 | 特殊计分逻辑边界漏洞

* 发现时间：2026-05-30
* 风险等级：中
* 触发入口：锁局结算 / 特殊计分 / 成绩归档
* 现象：
  * 某些特殊计分或锁局结算场景下，边界记录可能被错误纳入或排除。
  * 个别比赛会出现“明明像是有效记录，但结算口径不一致”的情况。
* 止损动作：
  * 对锁局赛参与排名的记录类型做硬过滤。
  * 将普通刷新记录与锁局有效记录分离。
* 排查过程：
  * 发现问题本质是 record 类型与结算口径没有完全对齐。
  * 某些“看起来像有效局”的记录并不属于应参与排名的集合。
* 修复动作：
  * 通过记录类型和会话绑定修正结算边界。
  * 需要保留的记录通过状态修正恢复，而不是按表面特征删除。
* 结果：
  * 该问题已归档，后续只在新赛事上线时保持口径警惕。
* 后续观察：
  * 新赛事上线前，先确认 record type 和结算规则映射关系。

## 2026-05-31 | 测试赛长期残留导致数据库脏数据

* 发现时间：2026-05-31
* 风险等级：中
* 触发入口：测试赛 / 成绩库 / 赛事状态管理
* 现象：
  * 某些已经结束或不应再参与的测试赛记录，长期留在数据库里。
  * 这些残留数据会干扰后续查询、补录和结算判断。
* 止损动作：
  * 梳理并修正测试赛的状态与归档口径。
  * 对历史残留数据做人工核对，必要时恢复为正确状态而不是直接删除。
* 排查过程：
  * 先从“为什么还有这些旧比赛”入手，再回头查状态与归档逻辑。
  * 确认不是单条记录异常，而是历史数据治理策略不够清晰。
* 修复动作：
  * 让查询和结算更严格地区分 active / cancelled / archived。
  * 对历史包袱进行人工清理和校正。
* 结果：
  * 已归档，不再作为当前持续关注项。
* 后续观察：
  * 后续建赛时要尽量避免测试赛长期残留为默认可见项。
