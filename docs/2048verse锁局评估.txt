2048verse 锁局评估（2026-05-02）

一、当前程序实际调用的接口

接口基地址：
https://backend.2048verse.com/leaderboard/user

请求形式：
?username=用户名&sort=date&variant=4x4&page=1&desc=true

二、实测返回字段

实测返回的 games 项里稳定可见的字段包括：
- id
- score
- played_at
- board
- info_flags
- variant

当前实测未见稳定返回：
- started_at
- created_at
- start_time

三、这意味着什么

1. 这条接口更像“用户历史对局列表”
2. 它不像“玩家开始一局”的事件接口
3. 如果只依赖这条接口，程序最多能识别：
   - 锁局后出现在列表中的下一条目标记录
4. 它不能可靠证明：
   - 玩家是否刚刚开始了一局
   - 某局是否在锁局时限内开始、但在更晚才结束

四、当前自动锁局能做什么

当前自动锁局适合做：
- 尽量自动抓到锁局后出现的下一条 Verse 记录
- 如果成功，减少手工补录量

当前自动锁局不适合做：
- 严格判断“玩家开始了一局”
- 作为唯一、绝对可信的锁局依据

五、现在最稳的使用建议

方案 A：把自动锁局降级为“自动识别下一条完赛记录”
- 主办方先点“开始锁定下一局”
- 如果后续正好抓到下一条 Verse 记录，就直接用
- 抓不到时走手动补录/手动绑定

方案 B：沿用更人工但更稳的旧思路
- 玩家先主动报“我要开始了”
- 主办方记录这个开始信号
- 玩家打完后提交成绩/截图
- 再按 ended_at + score 去做补录匹配和审核

如果追求现场稳定，当前更推荐方案 B。

六、当前结论

2048verse 现有这条 leaderboard/user 接口，不足以支撑“严格的开局锁定”。
它更适合作为赛后记录识别接口，而不是开局事件接口。
