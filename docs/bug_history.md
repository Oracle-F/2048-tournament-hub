## 2026-05-27 | Bot异常刷屏与账号风控警告

* 模块：QQ Bot / NapCat / 群聊接入
* 风险等级：高
* 现象：
  * 机器人在群聊和私聊场景里出现过高频响应、连续回复和复杂消息发送。
  * 账号侧出现过风控警告或疑似风险升高的迹象。
  * 当时一度需要优先止损，而不是继续扩大测试范围。
* 触发条件：
  * 群聊功能开放后，命令测试频率偏高。
  * 带 `@`、图片、文件、引用、长文本等复杂消息较多。
  * 平台侧对活跃度和消息形态更敏感。
* 根因：
  * 机器人在高风险场景下承担了过多主动交互。
  * 群聊与复杂消息会放大平台风控概率。
  * 高频测试叠加自动重试，容易让账号行为显得异常活跃。
* 修复：
  * 关闭群聊入口，回退到低风险可用态。
  * 取消默认 `@` 回复，减少主动打扰特征。
  * 暂停图片帮助、文件上传、成绩提交等高风险入口。
  * 只保留私聊轻命令作为最小可用面。
* 当前状态：观察中，当前维持保守策略，不恢复群聊和高风险入口。
* 相关文件：
  * `赛事中台/.env`
  * `赛事中台/docs/QQ账号风控止损与低风险恢复SOP_2026-05-30.txt`
  * `赛事中台/docs/项目现状总结_供新对话.txt`
* 备注：
  * 后续任何恢复都应遵循“先私聊、后群聊、先轻命令、后重功能”的顺序。
  * 这类问题的本质不是代码功能是否完整，而是账号稳定性优先级更高。

### 复盘

* 问题扩大原因：
  * 把群聊和复杂交互放得太早，测试节奏也偏密集。
  * 机器人在同一时段内同时承担查询、流程、上传和消息回复，行为特征过重。
* 当时排查过程：
  * 先确认是否为单纯代码错误，随后发现更像是平台风控与链路抖动叠加。
  * 观察到 `input_status`、回复超时和发送抖动并存，说明问题不只是逻辑层。
* 未来预防措施：
  * 默认保守开关，不把群聊作为常态能力。
  * 新功能优先私聊验证，再考虑是否开放到群聊。
  * 控制测试频率，避免连续轰炸式操作。

## 2026-05-28 | NapCat 偶发 Kick Offline / WebSocket 抖动

* 模块：NapCat / OneBot / QQ 消息链路
* 风险等级：中
* 现象：
  * 曾出现 `WebSocket closed by peer`、`sendMsg Timeout`、`get_msg timeout` 一类异常。
  * 有时消息能进来，但回复明显变慢，甚至无法及时返回。
  * 某些情况下会伴随掉线或连接重建现象。
* 触发条件：
  * reply / 引用消息较多时更容易出现。
  * 平台侧或客户端侧消息链路本身存在波动。
  * 处理过程里如果再叠加文件段补全、复杂消息段解析，延迟更容易放大。
* 根因：
  * NapCat / OneBot 链路不是稳定的同步 RPC，存在超时和抖动。
  * 处理 reply 时底层可能会去查消息详情，额外增加等待。
  * 链路异常会直接影响事件处理时延，进而影响用户体感。
* 修复：
  * 对发送超时和链路异常做降级处理，不再放大成群刷屏错误。
  * 尽量减少对 reply 信息的依赖。
  * 群聊默认不再依赖容易抖动的触发路径。
  * 对文件补全和回复发送增加超时容忍与日志记录。
  * 群聊里的意外异常会静默，不再把内部错误明文发给群友。
  * OneBot 的 reply 预查询默认关闭，避免 `get_msg` 反查把整条消息拖慢。
  * 发送层在超时或链路不稳时会退化为轻量文本兜底，减少完全无回复的概率。
  * 如果已经出现 `WebSocket closed by peer` 一类传输级异常，则不再继续补发兜底消息，避免把断链问题放大成二次消息噪音。
* 当前状态：观察中，已做降级和止损，但链路本身仍可能抖动。
* 相关文件：
  * `赛事中台/bot_private_qq/app.py`
  * `赛事中台/docs/项目现状总结_供新对话.txt`
  * `赛事中台/docs/QQ私聊bot接入说明.txt`
* 备注：
  * 这类问题不是单纯“代码 bug”，更像外部适配链路不稳定。
  * 后续维护应默认把它视为“可能随时抖动的基础设施”。

## 2026-05-29 | SQLite 锁竞争导致消息处理短暂繁忙

* 模块：数据库 / Bot / Hub 共享写入链路
* 风险等级：中
* 现象：
  * 私聊和群聊有时返回“系统繁忙，请稍后再试”。
  * Hub 的重刷新或其他后台任务运行时，bot 消息处理变慢。
  * 高并发阶段偶发数据库锁等待。
* 触发条件：
  * Hub 重操作与 bot 实时消息同时发生。
  * 一次消息处理里写事务持续过长。
  * 刷新、补录、评分、导入等重操作与查询叠加。
* 根因：
  * 项目主库是单个 SQLite 文件，并发写入脆弱。
  * 早期某些处理路径事务过长，锁占用时间偏久。
  * Bot 与 Hub 共享同一数据库，竞争更明显。
* 修复：
  * 增加 `busy_timeout` 和连接超时。
  * 尽量缩短事务范围，把网络请求和写库拆开。
  * 锁错误做短重试，并在用户侧降级为繁忙提示。
  * 让重刷新从“长事务”改为更细粒度的写入。
* 当前状态：已缓解，仍需持续观察高压时段。
* 相关文件：
  * `赛事中台/db.py`
  * `赛事中台/bot_private_qq/app.py`
  * `计分器/organizer_event_hub.py`
  * `赛事中台/docs/项目现状总结_供新对话.txt`
* 备注：
  * 这类问题一旦再升高，优先排查是否又出现了长事务或多个实例并存。
  * 若后续并发继续上升，可能需要重新评估单 SQLite 架构的上限。
