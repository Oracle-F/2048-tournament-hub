设计说明

一、总体思路

这个项目的核心思路不是追求“功能最多”，而是追求：

- 数据尽量可追溯
- 结算尽量一致
- Bot 尽量低风险
- 测试尽量能回放
- 文档尽量能接手

二、核心分层

1. 数据层
- SQLite 数据库
- 测试库固定使用 `data/testing.db`
- 重要快照和证据保存为独立文件

2. 业务层
- `services/` 目录
- 负责报名、提交、结算、刷新、查询、审核等逻辑

3. 接入层
- 主办方 CLI
- QQ bot
- 工具脚本

三、为什么强调“复用现有逻辑”

因为比赛系统最怕的是：

- 结算规则被重写
- 排名字段被改乱
- 历史数据无法再解释
- 回归测试没法继续沿用

所以遇到新需求时，优先在现有逻辑上补，而不是另起炉灶。

四、测试设计原则

- 测试案例尽量小而具体
- 新问题先变成最小复现
- 真实赛事快照优先保留
- API 快照优先保留
- 回放样本优先保留

五、文档设计原则

- 索引短
- 规则清晰
- 状态和历史分开
- 事故和归档分开
- 交接和操作分开

六、维护目标

让下一个接手的人能快速判断：

- 现在系统的安全边界在哪里
- 当前最重要的问题是什么
- 哪些东西已经稳定了，不用再重复折腾
