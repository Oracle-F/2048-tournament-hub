# 维护规则

这份文件说明：什么该记、写到哪里、什么时候归档、怎样保持项目长期可维护。

## 1. 先分清问题类型

### 需要写进 `bug_history.md` 的

- 现在还在发生的问题
- 可能复发的问题
- 会影响比赛结果、数据库、Bot 稳定性、账号安全的问题
- 需要后续 Agent 持续盯着的问题

### 需要写进 `bug_archive.md` 的

- 已解决
- 连续观察未复现
- 不再需要重点关注
- 只是历史线索的问题

### 需要写进 `incident_timeline.md` 的

- 影响范围较大
- 需要按时间顺序回溯
- 和账号风控、链路异常、锁竞争、结算偏差有关的事件

## 2. 写问题记录时的最小字段

建议至少写：

- 模块
- 风险等级
- 现象
- 触发条件
- 根因
- 修复
- 当前状态
- 相关文件

## 3. 先修复，再补回归

发现重要 Bug 时，优先级顺序是：

1. 先修根因
2. 再补 `tests/cases`
3. 再补真实快照或回放样本
4. 最后再考虑补文档说明

## 4. 真实快照与 API 快照

- `tests/real_tournaments/` 保存真实赛事快照
- `tests/api_snapshots/` 保存第三方 API 快照
- 这两类内容是长期资产，优先复用，不要轻易删掉或重写

## 5. 测试与数据原则

- 测试默认只用 `data/testing.db`
- 不要并行跑多个会重建 `testing.db` 的入口
- 不要碰正式数据
- 真实赛事回放、API 快照、历史案例尽量保持可复用

## 6. 文档写法

- 一个文件只负责一个职责
- 索引文件保持短
- 当前状态、历史归档、事故时间线分开写
- 不要把所有说明堆进一个大文档

## 7. Bot 与风控

如果问题可能影响账号稳定性、群聊风控或数据库锁竞争：

- 先止血
- 再恢复低风险能力
- 最后才考虑扩功能

## 8. 变更记录原则

- 不要主动扩大架构
- 改动尽量小而准
- 重要 Bug 修复后要留下回归样本
- 已归档问题不要重复写进 history
