# 维护规则

本文档用于约束项目历史文档的维护方式，目标是让后续 Agent 和新开发者能持续补写、归档和追踪问题，而不是只在当前对话里“记住”。

## 维护目标

* 记录项目曾经遇到过的风险、事故、异常和修复经验。
* 让新接手的人快速判断：当前要盯什么，哪些问题已经结束，哪些问题仍需观察。
* 避免单个问题记录无限增长，也避免同一事件反复重复写。

## 文档分工

* `bug_history.md`
  * 只保留当前值得持续关注的问题。
  * 适合记录仍在观察、高风险、可能复发、会影响比赛结果/数据库/Bot稳定性/账号安全的问题。
* `bug_archive.md`
  * 保存已解决、已归档、无需继续重点关注的问题。
  * 适合保留历史，但不再要求持续盯守的问题。
* `maintenance_rules.md`
  * 说明怎么写、何时写、如何归档。
  * 后续 Agent 接手时应优先查看。
* `agent_handoff.md`
  * 当前状态摘要和下一步建议。
  * 适合新对话快速接手。

## 写入规则

### 什么时候写入 `bug_history.md`

满足以下任一条件时，应该新增或更新 `bug_history.md`：

* 新出现的高风险问题。
* 可能复发的问题。
* 影响比赛结果、数据库完整性、Bot稳定性、账号安全的问题。
* 需要长期观察的问题。
* 已有问题出现新症状、触发条件变化、修复策略变化。

### 什么时候写入 `bug_archive.md`

满足以下任一条件时，应从 `bug_history.md` 迁移到 `bug_archive.md`：

* 问题已彻底解决。
* 连续观察一段时间未复现。
* 以后无需继续重点关注。
* 只是历史价值保留，不再影响当前维护。

### 什么时候只更新不新增

以下情况优先更新原条目，而不是新增相似记录：

* 根因一致，只是触发场景扩大。
* 同一问题换了表现形式，但本质仍相同。
* 同一事故在不同时间被再次确认。

## 记录方式

* 写清楚“发生了什么”。
* 写清楚“为什么发生”。
* 写清楚“怎么修的”。
* 写清楚“以后要注意什么”。
* 不写具体函数名和行号，避免文档和代码强绑定。
* 重点记录工程维护视角，而不是开发教程。

## 风险分级

* 高
  * 影响账号安全、比赛结果、核心数据完整性，或可能引发再次事故。
* 中
  * 影响稳定性、用户体感、局部流程正确性，但通常有缓解手段。
* 低
  * 只影响少量场景，或已彻底修复，仅保留历史参考价值。

## 归档规则

当 `bug_history.md` 中某条记录满足以下条件时，应迁移到 `bug_archive.md`：

* 已解决。
* 连续观察一段时间未复现。
* 当前无需继续重点关注。

迁移时应尽量保留：

* 标题
* 模块
* 风险等级
* 现象
* 根因
* 修复
* 相关文件
* 备注

但不需要保留详细复盘。

## 更新顺序

新 Agent 接手后，建议按照以下顺序处理：

1. 先看 `agent_handoff.md`。
2. 再看 `bug_history.md`，确认当前还要盯什么。
3. 需要了解历史时，再看 `bug_archive.md`。
4. 如果发现新的事故或修复结论，按本规则更新对应文档。

## 维护建议

* 不要让 `bug_history.md` 无限增长。
* 不要把所有历史都留在当前问题清单里。
* 不要因为问题已修好就立刻删掉历史，先归档再移出。
* 不要重复造记录，尽量沿用统一模板。
* 每次大修复后，最好立刻判断它是“继续观察”还是“可以归档”。
